# DBMS Weekly — 2026-06-22 (week of 2026-06-22–2026-06-28)

## PostgreSQL

PostgreSQL 19 is in beta (Beta 1 shipped Jun 4) and master is open for PG20, so this week's
ground-truth signal is in the commit stream — a mix of beta-period hardening of new PG19
features and fresh PG20 work. The committed highlights (links go to the commit):

- **[Inherited CHECK constraints can no longer be silently weakened](https://github.com/postgres/postgres/commit/0cd17fd)** — closes a data-integrity hole: a child table could relax a CHECK constraint it inherited from its parent, so the guarantee the parent advertised no longer held across the whole hierarchy. *(Andrew Dunstan · postgres git)* `[committed]`
- **[COPY TO (FORMAT json) now respects an explicit column list's order](https://github.com/postgres/postgres/commit/effb923)** — `COPY t (b, a) TO … (FORMAT json)` emits keys in the requested order instead of on-disk table order, matching how text/CSV already behave (FORMAT json is the PG18 addition). *(Andrew Dunstan · postgres git)* `[committed]`
- **[Temporal `FOR PORTION OF` gets two rough edges filed down for the beta](https://github.com/postgres/postgres/commit/a272a58)** — Peter Eisentraut moved the volatility check on the SQL:2011 `UPDATE/DELETE … FOR PORTION OF` period bounds into the planner, and separately made the executor [reject child-partition foreign tables under `FOR PORTION OF`](https://github.com/postgres/postgres/commit/a40fdf6) — hardening the new temporal-table feature before GA. *(Peter Eisentraut · postgres git)* `[committed]`
- **[`MERGE`/`SPLIT PARTITION` now honors `default_tablespace`](https://github.com/postgres/postgres/commit/cdae794)** — partitions produced by `ALTER TABLE … MERGE/SPLIT PARTITION(S)` were ignoring the session's `default_tablespace`, so reshaping a partitioned table could quietly land data on the wrong tablespace. *(Alexander Korotkov · postgres git)* `[committed]`
- **[Online data-checksum worker gets a correctness pass](https://github.com/postgres/postgres/commit/a4f02ca)** — Heikki Linnakangas tightened the background worker that enables/disables data checksums on a running cluster: more reliable distinction between worker invocations, no stale `worker_pid` left behind, and cleanup in `datachecksums_state.[ch]`. *(Heikki Linnakangas · postgres git)* `[committed]`
- **[`pg_stat_io` stops misattributing autovacuum-launcher extends](https://github.com/postgres/postgres/commit/4abf411)** — relation extends done by the autovacuum *launcher* were being counted like a worker's, skewing the per-backend-type I/O picture; Melanie Plageman's fix makes the accounting honest. *(Melanie Plageman · postgres git)* `[committed]`
- **[uuid→bytea cast marked leakproof](https://github.com/postgres/postgres/commit/6468f7a)** — a small but real planner win: a leakproof cast can be pushed below security-barrier views and RLS, where quals using it were previously held back for fear of leaking values. *(Masahiko Sawada · postgres git)* `[committed]`
- **[Runtime pruning + FDW executor fix in ModifyTable](https://github.com/postgres/postgres/commit/b43f8aa)** — when initial partition pruning drops result relations, the per-relation FDW state arrays are now re-indexed, so a foreign-partitioned `UPDATE`/`DELETE` can't read the wrong slot. *(Amit Langote · postgres git)* `[committed]`
- **[`pg_parameter_acl` nailed in the relcache](https://github.com/postgres/postgres/commit/4cc02b8)** — Jeff Davis marks the GUC-privileges catalog (the one consulted on `GRANT … ON PARAMETER` checks) as a nailed relcache entry, avoiding rebuild churn. *(Jeff Davis · postgres git)* `[committed]`
- **[Unsafe order of operations fixed in `ResourceOwnerReleaseAll()`](https://github.com/postgres/postgres/commit/ef01ca6)** — Tom Lane reorders resource-owner teardown so cleanup can't trip over partially-released state. *(Tom Lane · postgres git)* `[committed]`
- Also in-window: per-node I/O instrumentation got a [docs pass for `EXPLAIN (IO)`](https://github.com/postgres/postgres/commit/ed9ec3a) *(Tomas Vondra)*, the PG19 SQL/PGQ property-graph objects got [readable identity strings](https://github.com/postgres/postgres/commit/2a7e95b) *(Peter Eisentraut)*, and the [error for `RESPECT/IGNORE NULLS` on a non-window function](https://github.com/postgres/postgres/commit/419ce13) was made clearer *(Masao Fujii)*; plus null-pointer hardening in PL/Perl and PL/Python.

## Wider DBMS & distributed data

- **[ClickHouse 26.6](https://github.com/ClickHouse/ClickHouse/releases/tag/v26.6.1.1193-stable)** — the June monthly stable (`v26.6.1`, released Jun 25) leans hard into distributed and streaming analytics. *(ClickHouse · github.com)*
    - **Continuous queries** — streaming, incrementally-maintained queries over `MergeTree` tables.
    - **Multi-stage distributed execution** — MPP-style scatter/gather across worker nodes.
    - **`EXPLAIN WHATIF`** — test hypothetical skip indexes and estimate their skip ratio *before* building them (the optimizer-adjacent standout for an internals reader).
    - Geospatial pipeline in SQL: GeoJSON + Mapbox Vector Tiles output; `system.documentation` exposed as a queryable table; `FORMAT PNG` renders results as images in the terminal.
    - Behavior change: `insert_deduplication_version` default → `new_unified_hash`; insert dedup now works on the whole inserted block (per insert) rather than per part/partition.
    - Bug fix: data corruption writing Parquet to HDFS (`WriteBufferFromHDFS` wasn't flushing on `finalize()` since 26.1, truncating file tails including the Parquet footer).
