DBMS Weekly — 2026-06-07 (week of 2026-06-01–2026-06-07)

PostgreSQL

  • PostgreSQL 19 Beta 1 released — first beta of PG19 lands with a big feature set: in-core REPACK [CONCURRENTLY], the pg_plan_advice/pg_stash_advice planner-stabilization extensions ("plan advice", not "hints"), parallel autovacuum, online enable/disable of data checksums and logical replication without restart (effective_wal_level), SQL/PGQ property-graph queries, WAIT FOR LSN, INSERT ... ON CONFLICT DO SELECT, eager aggregation, JIT now off by default, and default_toast_compression=lz4. (postgresql.org)
  • Postgres 19 gets "plan advice" (query hints, sort of) — a walk through the new pg_plan_advice / pg_stash_advice contrib modules that let you capture and reapply planner decisions by query id, ending the project's long-standing "no hints" posture. (Shaun Thomas · pgEdge)
  • Your first graph query in PostgreSQL 19 (SQL/PGQ) — opens a series on the ISO SQL/PGQ syntax (CREATE PROPERTY GRAPH, GRAPH_TABLE) shipping in PG19, with worked examples over ordinary relational tables. (Hans-Jürgen Schönig · Cybertec)
  • When is a function leakproof? — Laurenz Albe digs into what LEAKPROOF really means for row-level security and security-barrier views, why it matters for RLS performance, and where the current definition is arguably too strict; framed as a request for comments. (Cybertec) [by committer]
  • pg_stat_statements: what it tells you, and what it can't — Radim Marek's two-part deep dive on why the same query splinters into many rows, how the fixed-size hash table evicts your tail under load, and why it counts rather than records — i.e. why it is not the query store you might want. (boringsql.com)
  • How the other half counts — Christophe Pettus contrasts how Postgres gathers planner statistics with the approaches taken by Oracle, Db2, MySQL, SQLite, DuckDB, and Snowflake. (thebuild.com)
  • data_checksums: a thirteen-year history still being written — uses the read-only data_checksums GUC as a hook into the long saga of page checksums, now finally toggleable online in PG19. (Christophe Pettus · thebuild.com)
  • Managed Postgres, examined: Azure Flexible Server — Azure's managed Postgres puts the standby in the commit path, so every write waits on synchronous replication to a second server — a real durability/latency trade-off competitors don't all make. (Christophe Pettus · thebuild.com)
  • A reviewer was born — Lætitia Avrot on the chronic shortage of patch reviewers: a clean, well-motivated io_uring fd-leak fix sat in the commitfest two months with zero reviewers, which is the norm, not the exception. (mydbanotebook.org)
  • Does pgBackRest work with pg_tde? — Stefan Fercot tests Percona's Transparent Data Encryption extension against pgBackRest and finds pgBackRest handles encrypted WAL segments transparently, contradicting an earlier claim that async archiving isn't supported. (pgstef.github.io)
  • pgsql_tweaks 1.0.3 released — Stefanie Janine Stölting's function/view bundle gets a maintenance release. (pgsql-tweaks.org)
    • tested against PostgreSQL 19 Beta 1
    • drops PostgreSQL 13 (EOL November 2025) from supported versions
    • get_markdown_doku_by_schema now uses COALESCE to avoid returning NULL on a NULL schema description

Migration experience

  • The night our tables wouldn't stop growing — a logical-replication migration where a 1-minute statement_timeout on the publisher kept killing the initial COPY; each restart left dead tuples behind, ballooning 50–90 GB source tables to 400+ GB on the subscriber overnight. Fix: ALTER ROLE replication_user SET statement_timeout = 0. (Semab Tariq · Stormatics)
  • File descriptors: the OS limit that takes down PostgreSQL — how fs.file-max and per-process RLIMIT_NOFILE exhaustion under high connection counts surfaces as a "database problem" that's really one layer down in the kernel, with the log sequence to recognize it. (Warda Bibi · Stormatics)

Research & cutting edge

  • GraftDB: dynamic folding of concurrent analytical queries — a multi-query execution engine that "grafts" a later-arriving query into a running one via shared operator state and per-query state lenses; TPC-H-derived workloads show up to 2.17x throughput and P95 latency as low as 0.17x an isolated-execution baseline. (arXiv) [paper]
  • LAANN: I/O-aware look-ahead search for disk-based ANN — a look-ahead search strategy for disk-resident approximate-nearest-neighbor indexes that overlaps and reorders I/O to cut latency on billion-scale vector search. (arXiv) [paper]
