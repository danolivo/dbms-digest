DBMS Weekly — 2026-06-22 (week of 2026-06-22–2026-06-28)

PostgreSQL mailing lists

  • [hackers] Conflict log history table for Logical Replication — error handling and a parallel-apply race get nailed down — the PG20-targeted feature that records apply-time replication conflicts into a system-managed table (conflict_log_destination = log | table | all, living in a pg_conflict schema) had its most intensive review week yet (Jun 22–26), and several open design questions were resolved. Highlights:
    • ERROR, not WARNING, on a failed conflict-log insert. Amit Kapila settled the long-running question: a failed insert into the conflict-log table now ERRORs out (attaching the conflict info as error context) instead of being silently downgraded to a WARNING — the WARNING path could spam the log on a persistent failure (e.g. disk full while writing WAL), let users miss conflicts, and required an "ugly" nested try/catch that was judged a future bug source.
    • A parallel-apply race that could lose the conflict row. Nisha Moond reproduced and fixed a race where, on an ERROR-level conflict, the leader's pa_wait_for_xact_finish() unblocks once AbortOutOfAnyTransaction() releases the worker's transaction lock, then raises "lost connection to the parallel apply worker" and SIGTERMs it mid-insert — losing the deferred conflict row. Fix adds a PARALLEL_TRANS_ERROR state plus an injection-point TAP test (039_pa_conflict_log_lock_wait.pl).
    • json vs jsonb, and stricter option parsing. The stored tuple uses json (chosen to preserve exact column order/formatting and avoid a per-conflict conversion), with the caveat that json has no ordering operator so ORDER BY replica_identity errors — flagged as needing more thought. Also tightened: drop the conflict-log table before its dependency records, and reject an empty-string conflict_log_destination to match every other subscription option.

(Nisha Moond, Amit Kapila, Dilip Kumar, vignesh C, shveta malik, Peter Smith · pgsql-hackers) [patch posted]
